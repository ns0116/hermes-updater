# Hermes Updater — 要求仕様書（PRD）

- 作成日: 2026-07-05
- ステータス: ドラフト（v0.1スコープ）
- 関連ドキュメント: 手動運用時代の詳細記録（Phase 14・Phase 16）。非公開のローカルメモに基づくため本リポジトリには含まれない（`CLAUDE.local.md`参照）

## 1. 背景・課題

2026-07-05、Hermes Agent（`C:\Users\<username>\AppData\Local\hermes`、ネイティブWindows版）が`pydantic-core`のバージョンドリフト（`2.47.0`がインストールされているのに`uv.lock`は`2.46.4`を要求）でクラッシュし、`hermes dashboard`/`hermes doctor`が起動不能になった。復旧の過程で、Hermes Agent本体（`hermes update`）とコミュニティ製Web UI（`hermes-webui`、別リポジトリ）の両方を最新化する必要が生じたが、その手順は：

1. Gatewayの停止
2. WebUIプロセス（ポート8788）の特定とUAC昇格による強制終了（`Stop-ScheduledTask`だけでは実プロセスが止まらない）
3. `hermes update`の実行
4. WebUIタスクの手動再起動
5. `hermes-webui`側は別リポジトリのため個別に`git pull`
6. `hermes doctor`/`/health`エンドポイントでの動作確認

という、手数が多く・手順を間違えるとGateway/WebUIの二重起動やアップデート失敗を招くプロセスだった（詳細は移行計画書Phase 14・16）。この手順を毎回人手で行うのではなく、**定期的に更新有無を検知し、ユーザーに通知したうえで、承認された場合にこの安全な手順を自動実行するツール**が必要。

### 1.1 hermes-webui自身の更新機能の調査結果（2026-07-05追加調査）

本アプリの設計に先立ち、`hermes-webui`自体に更新関連の機能がどの程度あるか調査した。結果、**かなり作り込まれた自己更新の仕組みが既に存在する**ことが判明し、これを踏まえて本PRDの方針を一部見直した。

- `GET/POST /api/updates/check` — WebUI自身とAgent双方の更新有無を30分キャッシュ付きでチェックするAPI（`api/updates.py`）
- `POST /api/updates/apply` — `git stash`→`git pull --ff-only`→`git stash pop`で適用。アクティブなチャットストリーム/実行中セッションがあれば更新をブロックする安全機構あり
- `POST /api/updates/force` — dirty/diverged状態向けの`git checkout .`→`git clean -fd`→`git reset --hard`
- **自己再起動の仕組み**: 更新適用後、バックグラウンドスレッドが猶予後に自分自身のプロセスを再起動する（Unix: `os.execv`、Windows: `DETACHED_PROCESS`で新プロセスをspawnし旧プロセスは`os._exit(0)`）。アクティブな処理の完了を待ってから再起動する
- Agent側の更新が成功すると`hermes gateway restart`を呼んでGatewayも連携して再起動する
- フロントエンド側には更新バナー・「Update Now」ボタン・LLM生成の"What's New"サマリー表示がある

**この調査を踏まえた設計方針への反映**（詳細は実装計画書「2.1 hermes-webui既存APIとの関係」参照）:
- **WebUI自身の更新**は、本アプリが生のgit操作をラップするのではなく、**WebUIが提供する`/api/updates/apply`をまず呼び出す方式を優先**する（stash/pop・競合検知・自己再起動まで既に安全に実装済みのため、車輪の再発明を避ける）。WebUIが応答不能な場合のみ、従来設計の手動git pull＋タスク再起動にフォールバックする
- **Agent側の更新は、引き続き`hermes update` CLI経由を正とする**。WebUIの`/api/updates/apply`はAgent側に対しても同じ「git stash/pull/pop」レベルの処理しか行っておらず、`hermes update`が本来行うuv依存関係同期・Node/npmビルド・スキル同期・config移行・Windows版Gatewayの一時停止/再開までは代替できない可能性が高い。これを見落とすと、今回まさに修正した「コードは新しいが依存関係が同期されていない」ドリフト事故を自ら再現しかねないため、Agent更新はWebUIの軽量パスに頼らず`hermes update`本体を必ず経由する
- WebUIの「更新中はアクティブなストリーム/実行中セッションをブロックする」安全パターンは、本アプリがWebUIプロセスを強制終了する前のマナーとして参考にする（必須の実装ではないが、望ましい配慮として将来拡張に記載）

## 2. 目的

- Hermes Agent / Hermes WebUIの更新有無を自動的・定期的にチェックする
- 更新がある場合、Windowsのトースト通知でユーザーに知らせる
- ユーザーが明示的に承認した場合のみ、実証済みの安全な手順（Phase 16準拠）で更新を適用する
- 実行内容・結果を常にログファイルに残す
- ログオン時に自動起動し、追加の手作業なしに常駐する

## 3. スコープ

### 3.1 スコープ内（v0.1）
- Hermes Agent本体の更新チェック・適用（`hermes update`をラップ）
- Hermes WebUI（`hermes-webui`リポジトリ）の更新チェック・適用（`git pull --ff-only`をラップ）
- Windowsシステムトレイ常駐UI（メニューからの手動操作一式）
- Windowsトースト通知
- ファイルへのログ記録（ローテーション付き）
- ログオン時自動起動（タスクスケジューラ登録）
- チェック間隔の設定（既定1日1回、変更可能な設計）

### 3.2 スコープ外（将来拡張として保留。詳細は「10. 将来拡張」）
- 完全無人（ユーザー承認なし）でのフル自動適用
- 更新失敗時の自動ロールバック
- 複数PC・複数ユーザープロファイルへの展開
- Discord/Slack等、Hermes gatewayを経由した通知（現状`gateway.platforms`は未設定のため対象外）
- GUIによる設定編集画面（v0.1は設定ファイル直接編集）

## 4. ユーザーストーリー

- ユーザーとして、PCにログオンしたら自動的に更新チェックが走ってほしい。手動でコマンドを叩きたくない。
- ユーザーとして、更新があるときだけ通知してほしい。何もなければ静かにしていてほしい。
- ユーザーとして、通知を見て「今は忙しいから後で」と流せる自由が欲しい（強制適用されたくない）。
- ユーザーとして、トレイアイコンから「今すぐチェック」「今すぐ更新」を能動的に行いたいときもある。
- ユーザーとして、更新が失敗したときに何が起きたかログで追いたい。無言で失敗してほしくない。
- ユーザーとして、このツール自体がHermes Agentのvenvに依存して共倒れするのは困る（今回のインシデントの再発防止）。

## 5. 機能要件

| ID | 要件 |
|----|------|
| FR-1 | 既定24時間間隔でHermes Agentの更新有無をチェックする。WebUIが正常応答する場合は`GET /api/updates/check`の結果を優先的に利用し（30分キャッシュ済みで軽量）、WebUIが応答不能な場合は`hermes update --check`相当の判定にフォールバックする |
| FR-2 | 既定24時間間隔でHermes WebUIの更新有無をチェックする。同様にWebUI自身の`/api/updates/check`を優先し、応答不能時は`git fetch` + `rev-list --count`にフォールバックする |
| FR-3 | チェック間隔は設定ファイルで変更可能（v0.1では設定ファイル直接編集、将来UIの設定画面を追加） |
| FR-4 | 更新が1件以上ある場合、Windowsトースト通知を表示する |
| FR-5 | トレイアイコンの右クリックメニューから「今すぐチェック」「状態表示」「更新を適用」「ログを開く」「設定」「終了」を提供する |
| FR-6 | 「更新を適用」実行時、対象ごとに以下の方式で処理する（詳細は実装計画書参照）: **WebUI**はまず`POST /api/updates/apply`（失敗/応答不能時のみ手動git pull＋タスク再起動にフォールバック）。**Agent**は必ず`hermes update --yes` CLI経由（Phase 16準拠: Gateway停止→WebUIプロセス停止→`hermes update --yes`→バージョン確認→WebUI再起動→`/health`確認） |
| FR-7 | 各ステップの実行結果をログファイルに記録する（タイムスタンプ・成否・出力を含む） |
| FR-8 | ログファイルはサイズ・世代数を制限してローテーションする |
| FR-9 | 通常ユーザーのログオン時に自動起動する（対話セッション、トレイ表示のため） |
| FR-10 | WebUIの更新がfast-forwardできない、またはWebUI自身の`/api/updates/apply`がconflict/dirtyを報告した場合は警告として記録し、Agent側の更新成功は損なわない（部分成功を許容） |
| FR-11 | UAC昇格をユーザーが拒否した場合、更新シーケンスをそこで中断し、次回ユーザーが手動で再試行できる状態を保つ |
| FR-12 | Agent更新のためWebUIプロセスを強制終了する前に、可能であればWebUIの「アクティブなストリーム/実行中セッション」有無を尊重する（WebUI側の安全パターンに倣った配慮。必須ではないが望ましい） |

## 6. 非機能要件

| ID | 要件 |
|----|------|
| NFR-1 | 本アプリのPythonランタイムはHermes Agentのvenv（`hermes-agent\venv`）に依存しない、独立した実行環境を持つこと（venv破損時にも本アプリ自体は動作できることを保証するため） |
| NFR-2 | Hermes Agentのvenvに対して`uv sync`等の直接操作は絶対に行わない。すべて`hermes`公式CLI経由で操作する |
| NFR-3 | UAC昇格はWebUI停止処理（1回）に限定し、必要以上に昇格プロンプトを出さない |
| NFR-4 | 設定・状態はJSON等の人間可読な形式で永続化し、障害調査時に手動確認できること |
| NFR-5 | 自動リトライは行わない（失敗時はログとユーザー通知に留め、再試行はユーザー操作起点とする） |

## 7. 成功指標

- ログオンのたびに手動操作なしで更新チェックが実行され続けること
- 更新適用を選んだ場合、全ステップの成否がログに残ること
- 「WebUIプロセスがロックしていて`hermes update`が失敗する」という既知の問題が、本アプリ経由の実行では発生しないこと
- Hermes Agentのvenvが壊れている状態でも、本アプリ自体（チェック・通知・ログ）は動作し続けること

## 8. 未確定・実装時に検証が必要な事項（スパイクタスク）

- `hermes update --check`の実際の出力フォーマット（"Update available: N commit(s) behind" / "Up to date"）は実装着手時に実機で再確認し、パース処理を確定する
- トースト通知ライブラリの選定: `win10toast`系は保守状況が不安定なため、実装時点で`win11toast`/`windows-toasts`等のメンテナンス状況を比較し確定する

## 9. 依存・前提

- `hermes`コマンドがユーザーのPATHに存在すること
- `git`コマンドがPATHに存在すること
- Hermes Gateway/WebUIの既存タスクスケジューラ登録（`HermesGateway-Native-Start`/`Hermes_Gateway`、`HermesWebUI-Native-Start`）がそのまま維持されていること（本アプリはこれらを前提に動作し、登録自体は変更しない）
- Windows 10以降（トースト通知APIの対象）

## 10. 将来拡張（v0.1スコープ外）

- 複数Windowsユーザー・複数プロファイル対応
- 外部通知（Discord/Slack等）との連携
- トレイメニューからの設定変更UI
